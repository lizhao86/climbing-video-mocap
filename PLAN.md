## 关键设计决策

### 1. 知识库 schema（knowledge_base/moves.json）

单 JSON 文件，每个动作：`id / name_zh(用书中译名) / aliases / category(沿用书的分类体系) / kind(posture静态|action动态) / book_ref(章节页码) / description / when_to_use / detectability(rule|rule_weak|vision|manual) / pose_rules / visual_cues_for_claude / common_errors / confusable_with`。

- **特征 = 命名特征 + 梯形隶属度区间**：condition 写 `{feature, range:[lo,hi], soft, weight, gate?}`，落区间得 1 分、soft 缓冲带线性衰减。读 PDF 只产出「哪个特征、什么区间、多重要」，规则引擎通用，加新动作零代码。
- `gate:true`（如可见度）不过 → confidence=null（遮挡≠不是该动作）。
- `side:either` 左右镜像各评一次取高者。
- 长度统一用现有 `body_scale`（肩中-髋中距中位数）归一化，与现有代码口径一致。
- posture 类带 `min_hold_s`（最短持续时长）。
- 教材定性为主：量化阈值由 Claude 定初值（书中物理讲解和图解做依据，note 记出处），再用老板视频调参 + 标注回流校准。**预期对齐：知识库是"读书起步、靠标注养准"。**

### 2. 动作分段（climb_segments.py）—— 2026-07-16 检测核心改为锚点位移法 v2

原计划的「速度滞回阈值法」（allspeed=max(四肢速度) + mean±kσ 滞回）**已实测否决**：
对慢速换点有结构性盲区——IMG_6952 的 14-20s 慢速横移整段漏检，速度可以一直低于
任何合理阈值，但位移必然累积。与全程 3fps 抽帧人眼直读对照后换方案。

现行方案（锚点位移法）：
- 每关节（四肢端点+双膝）在 0.5s 滑窗内位移 > D_MOVE=0.3 身长 → 该帧"换点中"
- **两层输出**：① timeline 段（六关节并集→合并/去短→move/static，给节奏统计）；
  ② **肢体换点事件**（每一手/每一步：limb+时间段+位移向量，S4 动作识别的基本单元）
- 事件过滤对人眼时间轴 18 窗口扫参定值（EV_MIN_DEV=0.4bl、EV_MIN_DUR=0.3s、同肢体间隙 0.5s 合并）：40 事件零漏检
- 左右判定：MediaPipe 标签默认可信（12/15 段正确），扭身/遮挡的错误段无启发式可救
  （镜像互换/几何位置/肩朝向/链距离四方案均实测否决），走 `annotations/<base>_side_overrides.json`
  人工覆写（source=manual），深度扭身段自动标 side_confidence=low
- 首尾 0.25s 速度边缘伪影钳制（平滑+梯度在边界造出假 19bl/s）
- **static 段同样输出**——posture 类动作（heel hook、锁定、休息、旗式）发生在静止段，是识别主战场
- 旧 count_moves 口径保留在 metrics.json 不动
- 审阅工具：climb_segments_review.py 生成烧字幕审阅视频（实时显示"出X肢"+✎人工/？低置信）

### 3. 特征注册表（features.py）

帧级（posture 用，左右成对、身长归一化、y 向上为正）：`ankle_rel_hip_y、knee_rel_hip_y、wrist_rel_shoulder_y、knee/elbow/hip_angle、heel_leads_toe_x、foot_pitch、ankle_cross_midline、feet_spread、hands_gap、wrists_crossed、shoulder_width_ratio（侧身深度代理）、hip_wall_proxy、torso_lean、*_visibility`。
段级（action 用）：主导肢体、位移方向幅度、peak_com_speed、双手同时脱点、段前后髋高变化、段时长。
注：pose2d.csv 已含 heel / foot_index 点，挂脚/toe hook 判定可用。

### 4. 2D 可判性结论（写进知识库 detectability 字段）

- **rule 可判**：高跨步、挂脚 heel hook、直臂悬挂、锁定 lock-off、蹲跳 lunge/dyno、同点 match、交叉手、外撑 stemming、rockover、抖手休息
- **rule_weak 需视觉确认**：挂旗 flag（内/外旗区分靠视觉）、垂膝 drop knee、侧身/twist-lock、toe hook、下壓 mantling、deadpoint
- **vision/manual（阶段三）**：手型 13 种（无手指关键点）、smearing、锁膝 knee bar、campus

### 5. 置信度分层（阈值放 climb_match.py 顶部，验证时用 IMG_6952 调参）

top1 分数 s1、与 top2 差距 margin：
- **高**（s1≥0.75 且 margin≥0.20 且 rule 类）→ 直接采纳 `source:"rule"`
- **中**（0.45≤s1<0.75 或 margin<0.20 或 rule_weak/vision 类）→ 截 3 帧（段起/峰值/段末，原视频帧 + 骨架对照版）→ Claude 会话内 Read 判定，prompt 附规则命中明细 + visual_cues + 案例库 few-shot 图
- **视觉后仍中** 且段时长≥1s 或近 crux → 进人工标注队列
- **低**（s1<0.45 视觉也无候选）→ move 段标 `basic_move`、static 段标 `rest_or_adjust`，不打满屏 unknown

### 6. 标注 HTML（climb_annotate_html.py 生成）

单文件自包含，关键帧 base64 内嵌（每段 3 帧 480px JPEG）。每段一屏：3 帧可放大、Top-3 候选按钮（含分数+命中条件中文解释）、「其它动作…/普通移动/跳过」、备注框；快捷键 1/2/3/0；localStorage 防丢；底部「导出标注」Blob 下载 `<base>_annotations.json`（兜底：复制剪贴板）。老板导出后放 `annotations/` 目录，跑 merge 脚本回流。

### 6.5 已否决方案：教材卡通图跑骨架建 pattern（2026-07-15 实测否决）

MediaPipe 在教材手绘插图上实测：7 个用例（整页+单人裁剪）仅 2 个检出且关节点明显错位（髋部飘到背部等），单人裁剪图反而全部检不出。结论：**量化 pattern 只能来自真实视频的标注回流（见 7. 案例库），书图只做人看 + Claude 视觉判定的参考图**。不要再提议在卡通图上提取骨架特征。

### 7. 案例库（cases/cases.json + cases/keyframes/）

跨视频累积。每条案例：label、side、source(manual>vision)、时间段、**全量特征向量快照**、关键帧路径、老板备注。
- **kNN 校正（规则层自动）**：z-score 欧氏距离 k=3；近邻一致 → 置信 +0.15（封顶 0.95）；与规则 top1 冲突且距离近 → 强制进视觉裁决；案例 <5 条静默不生效
- **few-shot（视觉层）**：视觉判定时附 1-2 张最近邻已确认关键帧

### 8. 新版报告卡（climb_report_v2.py，v1 保留，分两步交付）

**v2a 节奏统计（不依赖识别，S3 交付）**：
- 完成时间：起攀（COM 首次持续上升 >0.3 身长）→ 完攀（COM 达全局最高后保持 >1.5s）
  - **2026-07-16 补「上墙门」**：原规则在含"走向岩壁"的视频上必然把起攀判到 t=0。
    相机静止，人走向岩壁=在**深度方向**远离相机=画面里变小且上移，2D 重心高度被透视
    被动抬高。IMG_6942 实测：走近的 2s 里 COM"上升"1.09bl（0.55bl/s），真正爬墙 25s
    才升 2.5bl（0.1bl/s）——**走路比爬墙"升"得还快**，任何基于速率/位移的规则都分不开，
    全局最低点必然落在走路起点。解法：先用**深度无关**的身体自身相对量
    （同一深度两点之比不受距离影响）判断是否已搭上岩点——较高那只手腕高过肩线且持续
    0.5s——再从该时刻起套用 COM 规则。IMG_6942 起攀 0.0→1.7s（画面 1.5-1.6s 搭点，吻合）；
    IMG_6947 判 0.0s（该视频一开始人就在墙上，正确）。
    ⚠️ 已否决：用每帧表观肩髋距探测走近阶段——被躯干旋转的 ±20% 波动淹没，实测不可用。
- 切换耗时：每 move 段前 static 段时长 = 准备/犹豫时间，条形图 + 中位数
- 难点：准备时长 >中位数×2.5，或同高度区间反复 move ≥3 次；与 v1 crux 互证，双命中标「确认难点」
- 休息/攀爬占比：static 再分 rest（直臂+低速+>2s）/ adjust，三段占比
- 弯臂检测：静止段肘角持续 <150° 超 2s = 弯臂耗力（高置信硬指标，先只上这类）
- 休息点质量：直臂(+)、脚高髋低(+)、难点前(+)、弯臂(−)、过长>8s(−)

**2026-07-16 S3 实现要点（写死在 climb_report_v2.py 顶部旋钮，验收后可调）**：
- **可见度门槛必须按左右臂各自独立判定，且要低（VIS_MIN=0.3）**。攀岩时手抓在岩点上
  贴着墙，MediaPipe 给手腕的 visibility 长期很低（实测中位数 0.19-0.38）。
  用"左右肘+左右腕四点同时 >0.5"的与门只剩 **9%** 的帧可用，rest/弯臂必然全灭；
  改单臂门槛 0.3 后"至少一条臂可用"回到 **77%**。肘角取自 world landmarks，
  已抽帧人眼核对低可见度下依然准（3.8s 直臂 167°、4.2s 弯臂 94° 与画面一致）。
- **可见度闪断要补洞**（VIS_GAP_FILL_S=0.5）：0.2s 的瞬时闪断会把连续弯臂段切碎导致
  达不到 2s 阈值而漏检（实测 4.6s 的弯臂段被切成 2.1s）。
- FOOT_HIGH=0.80：髋-踝落差中位数实测 0.9-1.0bl，取 25 分位才是"明显比平时踩得高"。
- **🚨 v1 的 crux（吃力点）实测退化成了「关节弯得深」排序，不是「用力」排序**
  （2026-07-17 拆解，起因：老板看着 14.4s 那张卡说「图里人明明在放松，不剧烈」）。
  v1 原式：`强度 = z(重心加速度) + z(肢体速度) + (120-膝角)/40 + (120-肘角)/40`。
  IMG_6942 全片四项的实际分布：

  | 项 | 中位数 | 90 分位 | 最大 |
  |---|---|---|---|
  | z加速度 | **-0.12** | **0.04** | 28.55 |
  | z速度 | **-0.26** | 0.73 | 13.70 |
  | 膝屈项 | 0.48 | 1.44 | 2.08 |
  | 肘屈项 | 0.61 | 1.43 | 2.69 |

  两个 z 分数被极端离群值撑大的标准差压扁，正常帧几乎恒为 0；**屈曲项才是实际主导**。
  实测 5 个 crux 的主导项：3.1s=加速度(62%)、7.3s=膝屈(52%)、9.8s=膝屈(66%)、
  14.4s=速度(47%，且总分最低)、37.6s=肘屈(63%，还在完攀之后)。
  **攀岩时膝盖弯 78° 完全可以是舒服地踩在脚点上**——把屈曲等同于发力在本领域不成立。
  处理：报告卡不再声称「动作剧烈」，改为如实报出每处的主导项（见 climb_report_card.py）。
  ⚠️ **S8「动作缺陷」若打算复用 v1 的 crux，先读这条**；真要「发力」信号需另设计
  （建议：加速度用分位数而非 z 分数，避免被离群值压扁）+ 抽帧验证。v1 脚本仍冻结不改。
- **❌「双命中=确认难点」已废除，改为两类难点并列展示**（老板 2026-07-17 拍板）。
  IMG_6942 实测双命中 **0**，抽帧核对确认 **v2a 判对了**（19-24s 人确实卡在同一高度弯臂
  硬扛），v1 五个吃力点无一命中该段。原因：两者测的是**正交的现象**——
  · **v1「发力型」**：重心加速度 + 肢体速度 + 关节极端屈曲 → 找"拼尽全力"的瞬间；
  · **v2a「卡住型」**：停顿异常长 / 同高度反复出手 → 找"不知道怎么办"的时刻。
  真正的难点常常是后者，而 v1 结构上看不见它。硬套互证只会把真难点标成"疑似"。
  现在两类各自标 source（`v2a` / `v1`）并列列出，`v1_nearby_t` 仅记录时间邻近关系供参考、
  不影响可信度。实测两种情况都存在：IMG_6152 的 12-13s 卡住型与 v1 的 13.4s 邻近（既难想
  又难拉），IMG_6942 的两类则完全错开（费力的地方和想不明白的地方是分开的）。
- **「同高度反复出手」必须加最小横跨时长**（REPEAT_MIN_SPAN_S=4.0）：不加会在**起攀处
  必然误报**——起步几手的中点高度天然都落在最低那个箱里。IMG_6152 实测：起步 0.23-3.23s
  出手 3 次但 3s 爬了 0.67bl（正常起步，滤掉）；9.2-17.37s 出手 3 次却 8.2s 只升 0.8bl
  （真卡住，保留）。detail 里同时报出"横跨 Xs 却只上升 Ybl"，让判据自解释。
- **顶点保持要报实际时长而非只报 True/False**：IMG_6152 保持 1.37s vs 阈值 1.5s，其实是
  正常完攀后下撤，报 False 会误导成"可能没到顶"。现在 metrics_v2.json 出 `top_hold_s`，
  报告卡按"≥60% 阈值"给不同措辞。
- **🚨 肘角必须过两道独立体检：可见度 + 解剖学**（ANAT_LO/HI=0.7/1.4，2026-07-17 加）。
  起因：老板说「14.4s 图里右臂看着挺直的」。查下去发现三件事：
  1. **单目在 2D 上分不出"直臂"和"朝深度方向弯的臂"**——直臂投影是直线，朝相机弯的臂
     投影也是直线。14.4s 右臂 2D 量角 170.7°（看着直）、3D 量角 111.6°（弯）。
     判据：**直臂的上臂与前臂朝同一 3D 方向，投影缩短比例相同，2D 的前臂/上臂比应 ≈1.0**；
     实测 0.50 → 两段朝向不同 → 肘部确实有弯，只是弯向深度方向。老板的眼睛和 2D 数据
     面对同一张投影，有同样的盲区——**这类争议不能靠看图裁决，要靠几何**。
  2. **MediaPipe 的 3D 重建约一半时间在解剖学上就是不可能的**：前臂/上臂比在
     IMG_6942/6152 上只有 47-63% 的帧落在 0.7-1.4，95 分位达 2.0-2.9（前臂比上臂长 2-3 倍）。
  3. **可见度分数预测不了重建质量**：15-20% 的帧可见度达标却解剖学不可能。两者独立，必须
     都过。实测这道门槛把一个被烂重建撑到 149°（差点够到直臂线 150° 被判成"休息"）的
     静止段打回 123° 真值；弯臂段 19.3-22.8s 则从 3.41s 收紧到 2.04s，结论存活。
  报告卡须把「两道体检都过的帧占比」露出来——它决定了弯臂/休息这两项结论有多硬。
- **rest 分支在三条素材上全部未触发**，但**「从不直臂休息 → 停顿全在弯着扛」是过度推断，
  已废除**（老板 2026-07-17 顶回）。逐段看是**分裂**的：IMG_6942 有 3 次停顿明确弯着
  （81-99°），另 4 次手臂接近伸直（123-143°）；IMG_6152 甚至有停顿达 166°（完全直臂），
  只是都短于 REST_MIN_S=2s 而没被计为 rest。**成立的只有「没有检测到 ≥150° 持续 2 秒的
  直臂休息」这一句**，不能外推成"全程弯着扛"。
  ⚠️ 更严重的是当时还写了「这是**前臂先泵**的直接原因」——**老板从没说过他前臂泵，这是
  凭空捏造的症状 + 倒推的因果**。教训：指标只说指标测到的东西，不要发明用户没报告过的
  现象来给结论加戏。
  副作用：休息点质量（第 6 项）实现了但未被任何素材验证，需一条含真实休息的视频。
  注：抱石线路短而连续，真休息本就是先锋/长线才有的东西，若后续素材仍全是抱石，此项
  可能长期无法验证——届时考虑放宽 REST_MIN_S 或改口径。

**报告卡合并（climb_report_card.py，2026-07-17 老板拍板）**：
一条线只出**一张**「攀岩报告卡.html」，v1/v2a 是开发阶段代号，不该漏到用户面前。
- 架构：v1/v2a 的脚本都不改——`metrics.json` / `metrics_v2.json` 就是它们的产出契约，
  报告卡只做展示层，读两份 metrics + segments + CSV + 原片。
- v1 的 HTML 是冻结脚本的必然副产品，`--out` 指到 `数据/v1原始/` 眼不见为净。
- 设计：暗色·影像优先。图表全部**内联 SVG**（可主题化 / 任意缩放清晰 / 与 HTML 精确对齐 /
  彻底摆脱 matplotlib 的中文字体依赖）；难点截图按骨架包围盒**自动取景**成 4:5 特写；
  时间线的三角标记与弯臂段**可悬浮浮出当时画面**（纯 CSS，热区按百分比定位——
  实测 1440/885 两种宽度下与 SVG 图元偏差均为 0px；**用 px 会错位**，SVG 有缩放系数）。
- 已砍：重心轨迹图、四肢活动图（偏技术，对复盘无用）。
- ⚠️ **写图不能用 `cv2.imwrite`**：Windows 上遇到非 ASCII 路径**静默失败**（返回 False、
  不抛异常、不留文件）。v1 的 climb_analyze_report.py 就栽在这——它的 5 张 crux 截图
  从来没落盘，报告卡里一直是裂图。改用 `cv2.imencode` + Python `open()`，且写不成就抛异常。
