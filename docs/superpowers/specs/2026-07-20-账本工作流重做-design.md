状态：设计已定案（老板逐节确认）

## 起因

J1 上线了网页填写表单（`climb_journal_edit.py` + `打开账本填写.bat`），
让老板在浏览器里填难度/完攀/岩馆。**7 条素材的 `线路.json` 全是空的，一条没填过。**

### `线路.json` 字段改版

现有 7 个 sidecar 全空，改字段名零成本。

```json
{
  "类型": "抱石",
  "难度": "V4",
  "完攀": true,
  "地点": "岩时攀岩馆",
  "线路名": "",
  "日期": "2026-03-29",
  "备注": "",
  "视频指纹": "sha256:ab12…",
  "登记时间": "2026-07-20 14:32"
}
```

改动点：

- 「岩馆」→「**地点**」：室内外通用，野外岩场也填这里。
- 新增「**类型**」：`抱石` / `顶绳` / `先锋`。
- 新增「**视频指纹**」：判重依据，Claude 自动算，老板不管。
- 新增「**登记时间**」：元数据写入时刻，便于排查。
- 「线路名」保留但只有野外才填（室内线路没有正式名字）。

### 难度双刻度（关键约束）

**抱石用 V 级，顶绳/先锋用 5.x（YDS），两套刻度不能混一个榜。**
聚合与展示都必须按「类型」分开统计：首页荣誉行出两个数字
（「抱石最高 V5」+「绳索最高 5.10b」），难度分布条也分两组。

解析规则：`V\d+` 判为抱石刻度，`5\.\d+[a-d]?` 判为绳索刻度。
两者都不匹配时该条不进难度榜（但仍进里程/足迹统计），并在账本页标「难度未识别」。

### 个人数据

`账本配置.json` 加 `体重_kg`。

- `身高_m` 有实际用途：把 `net_gain_bl`（身长）换算成爬升米数。
- `体重_kg` **只存不用**，不参与任何指标。体重在攀岩里唯一能算的是「做功焦耳数」，
  那是发明出来的数字，不是老板会看的东西（CLAUDE.md「不许发明老板没报告过的症状」）。
  存着是因为老板要求固化个人数据，将来若出现真实用途再接。

## 二、分析流程

老板丢视频进 `收件箱/`，说一句「分析」，Claude 执行：

1. **扫描判重**：对收件箱每个视频算指纹，与所有 `素材/*/线路.json` 的「视频指纹」比对。
2. **重复命中**：当场报「这条 3 月 29 日分析过，V4 已完攀」，问老板跳过还是重跑。
   重跑 = 覆盖该条产物，保留原 `线路.json` 元数据。
3. **新视频**：后台开跑流水线（`run_in_background`）。
4. **并行追问**：流水线跑的同时，Claude 在对话里问 5 项——
   类型 / 难度 / 完攀 / 地点 /（野外才问）线路名。答完流水线也差不多跑完。
5. **归档**：视频与产物落到 `素材/<片名>/`，Claude 直接 Write 写 `线路.json`。
6. **聚合刷新**：重跑 `climb_journal.py` + `climb_journal_card.py`。
7. **汇报**：结论先行——这条爬了多久、升了多少、难点在第几秒。

### 判重指纹

`sha256(文件大小字节 + 文件头 8MB)`。改文件名认得出；重新编码/裁剪认不出（可接受，
那本就是另一个文件）。选头部而非全文是因为原片可达数百 MB，全文哈希拖慢扫描。

### `climb_intake.py`（新脚本）

只做两件事，**不算任何指标**（分层硬约定）：

- `scan`：列收件箱视频 + 指纹 + 判重结果，输出 JSON 给 Claude 读。
- `run <视频>`：跑完整流水线到 `素材/<片名>/`。

顺手固化三件老板不该操心的事：

1. **Windows 三条铁律**写死在脚本内：`PYTHONUTF8=1`、`PYTHONIOENCODING=utf-8`、
   `CLIMB_ROTATE=0`、调用 `C:\venvs\climb310` 的 ASCII 路径 Python。
2. **远景素材自动提示**：标准版跑完检出率 < 50% 时提示改用 `climb_pose_roi.py`
   （不自动切换——ROI 需要播种参数，得人判断）。
3. **跟丢校验**：跑完检查骨架连续性，异常时提示人工核对标注视频
   （检出率高 ≠ 跟对了人，见 CLAUDE.md 验证素材条款）。

流水线顺序：
`climb_pose.py` → `climb_analyze_report.py` → `climb_segments.py` →
`climb_report_v2.py` → `climb_match.py` → `climb_report_card.py`

元数据**不经过脚本**，由 Claude 直接 Write `线路.json`，少一层转译。

## 三、账本页

两层，都是自包含静态 HTML，双击即开，**不再需要起 127.0.0.1:8765 服务**。

### 首页 `攀岩账本.html`

上半屏三行并列（老板选定：不分主次）：

| 行 | 内容 |
|---|---|
| 荣誉 | 抱石最高 V5 · 绳索最高 5.10b · 各难度完攀数分布条 · 本月最难 |
| 里程 | 累计爬升 XX 米 · 在墙 XX 小时 · XX 次出手 · 爬岩 XX 天 |
| 足迹 | 去过 X 个地点 · 最长连续 X 周 · 动作收集（高跨步 22 · Match 5 · 换手 4） |

下半屏目录：按月分组、日期倒序。每行 `日期 · 地点 · 类型 · 难度 · 完攀标 · 时长 · 爬升`，
点击进对应报告卡。

动作收集沿用现有口径（conf ≥ medium，即 high_step / match / hand_change 三种）。
连续周数沿用现有双口径（最长连续 + 当前连续，断了就是 0，不粉饰）。

### 详情页 `素材/<片名>/攀岩报告卡.html`

现有报告卡原样收编，**内容一个字不改**。只在顶部加一条页眉：
「← 返回账本」+ 这条线的难度 / 地点 / 完攀 / 日期。

## 五、分层约定合规

| 层 | 文件 | 职责 |
|---|---|---|
| 取数 | `climb_pose.py` / `climb_pose_roi.py` | 视频 → CSV（v1 不动） |
| 指标 | `climb_analyze_report.py` / `climb_segments.py` / `climb_report_v2.py` / `climb_match.py` | → 各自的 json |
| 编排 | `climb_intake.py`（新） | 判重 / 跑流水线 / 落位，**不算指标** |
| 输入 | `线路.json` | Claude 对话追问后写入 |
| 聚合 | `climb_journal.py` | 扫 json + sidecar → `journal.json` |
| 展示 | `climb_journal_card.py` / `climb_report_card.py` | 只读 json → HTML |

「改样式不碰指标，改指标不碰样式」继续成立。v1 两个脚本不动。

## 验收标准

1. 往 `收件箱/` 丢一个新视频，说「分析」，全程对话完成录入，账本页出现新记录。
2. 同一个视频改名后再丢一次，被判重拦下并报出原记录信息。
3. 7 条旧素材经对话逐条补齐元数据后，首页三行数字与素材清单人工核对一致。
4. 首页每条记录可点进对应报告卡，报告卡可点回首页。
5. 全程不需要启动任何本地服务。
